## What This Video Demonstrates

A fully integrated personal AI operating system: Hermes Agent hosts named personas (the Pantheon system), reads and writes to an Obsidian vault for persistent memory, scrapes B2B leads via Apollo.io, integrates Gmail and Calendar through Zapia MCP, and runs a nightly Claude Code bridge that generates a morning improvement brief from overnight chat logs.

## Workflow Steps

1. Define named personas in Hermes: Labyrinth (deep research), Mercury (autopilot/cron tasks), Philosopher (reasoning and reflection) — each has its own model assignment, role prompt, and behavioral constraints
2. Write a `soul.md` file: a brain dump of agent personality, values, and identity that all personas reference
3. Connect Hermes to an Obsidian vault — Hermes reads from and writes to vault notes, giving agents persistent memory that survives session restarts
4. Configure Zapia MCP for Gmail and Calendar with principle-of-least-access: draft-only mode (no send permission granted)
5. Set up Apollo.io API integration — demo: query 20 roofing companies in Austin TX, results returned as structured lead data
6. Mirror Hermes config to a private GitHub repo as backup and version control
7. Configure a cron job (Mercury persona): overnight, read Claude Code chat logs → generate a morning improvement brief summarizing patterns, errors, and suggestions
8. Each morning, review the brief generated by the Claude Code OS bridge before starting work

## Claude Code Integration Points

- **OS bridge (core use case)**: Claude Code chat logs are the raw input — a cron job reads them nightly and generates improvement briefs; this is a direct Claude Code integration
- **MCP servers**: Zapia MCP is already an MCP integration; any Claude Code session can consume the same MCP tools Hermes uses, unifying the toolset across environments
- **Skills**: The Pantheon personas map to Claude Code skill roles — Labyrinth's deep research behavior could be a `/deep-research` skill, Mercury's cron behavior a `/schedule` skill
- **CLAUDE.md**: `soul.md` is equivalent to a project-level CLAUDE.md — the same behavioral constraints could live there for use in Claude Code sessions
- **Hooks**: A `Stop` hook in Claude Code could write a session summary to the Obsidian vault automatically at session end, feeding the same memory system Hermes reads from
- **GitHub sync**: Claude Code's git tools can push Hermes config changes to the private mirror repo as part of any session

## Key Insight

**Treating your AI agent system like an operating system** — personas as apps, Obsidian as persistent storage, cron as the scheduler, GitHub as version control — gives each component a clear, non-overlapping role and makes the whole system maintainable and extensible.

## Reusable Primitives

- Pantheon pattern: named personas with distinct model/role/prompt assignments inside one agent shell
- `soul.md` as a shared identity file that all personas reference — single source of truth for agent behavior
- Obsidian vault as a persistent, human-readable memory layer that survives session restarts
- Principle-of-least-access for MCP integrations: grant draft-only, read-only, or scoped permissions — never full send/write access by default
- Claude Code OS bridge: read your own chat logs with a cron job to generate self-improvement feedback
- Private GitHub repo as config backup and rollback mechanism for agent setups

## Limitations / Gotchas

- Pantheon system is Hermes-specific — persona assignments and session locking do not transfer directly to other agent frameworks
- Apollo.io API access has its own cost and rate limits; lead data freshness depends on Apollo's database, not your scraping
- Zapia MCP is a paid third-party integration layer — another vendor dependency in an already complex stack
- Obsidian as memory works well for human-readable notes but is not a queryable database; large vaults may require search/indexing workarounds
- The Claude Code OS bridge depends on chat log format stability — if Claude Code changes its log structure, the cron job breaks
- Video demonstrates the architecture and tooling but does not show failure handling or what happens when any layer in the stack goes down
